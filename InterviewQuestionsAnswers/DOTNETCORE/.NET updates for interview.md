## ✅ Step 2: Define Interfaces (VERY IMPORTANT)

Interviewers love **interfaces + DI**.

```csharp
public interface IOrderService
{
    Task<Order> CreateOrderAsync(CreateOrderRequest request);
}

public interface IPaymentService
{
    Task<bool> ProcessPaymentAsync(Payment payment);
}

public interface IInventoryService
{
    Task<bool> ReserveStockAsync(List<OrderItem> items);
}
```

✅ Shows:

*   Loose coupling
*   Testability
*   Clean architecture

***

## ✅ Step 3: Domain Models

```csharp
public class Order
{
    public Guid Id { get; private set; }
    public List<OrderItem> Items { get; private set; }
    public OrderStatus Status { get; private set; }
    public decimal TotalAmount { get; private set; }

    public Order(List<OrderItem> items)
    {
        Id = Guid.NewGuid();
        Items = items;
        Status = OrderStatus.Created;
        TotalAmount = items.Sum(i => i.Price * i.Quantity);
    }
}
```

✅ Entity rules live **inside** the domain model.

***

## ✅ Step 4: Service Implementation

```csharp
public class OrderService : IOrderService
{
    private readonly IPaymentService _payment;
    private readonly IInventoryService _inventory;

    public OrderService(
        IPaymentService payment,
        IInventoryService inventory)
    {
        _payment = payment;
        _inventory = inventory;
    }

    public async Task<Order> CreateOrderAsync(CreateOrderRequest request)
    {
        var order = new Order(request.Items);

        var stockOk = await _inventory.ReserveStockAsync(order.Items);
        if (!stockOk)
            throw new Exception("Out of stock");

        var paymentOk = await _payment.ProcessPaymentAsync(
            new Payment(order.TotalAmount));

        if (!paymentOk)
            throw new Exception("Payment failed");

        return order;
    }
}
```

✅ Interview keywords:

*   **Single Responsibility**
*   **Dependency Injection**
*   **Async/await**

# PART 2️⃣ Saga Pattern – FULL DETAILED EXPLANATION

This is **critical**. Many candidates fail here.

***

## 🔥 What Is the Saga Pattern?

> **Saga Pattern** is a way to manage **distributed transactions** across multiple microservices **without using a single database transaction**.

✅ Core idea:

*   Long‑running process
*   Each step is atomic
*   Failures are handled using **compensating actions**

***

## ❌ Why Traditional Transactions Don’t Work

In microservices:

*   Each service has its **own database**
*   ACID transactions across services are **not possible**

✅ Hence → **Saga Pattern**

***

## ✅ Real Example: Order Placement

### Steps:

1.  Create Order
2.  Reserve Inventory
3.  Process Payment

❓ What if payment fails after inventory is reserved?  
→ **We must undo inventory reservation**

***

## ✅ Saga Pattern Types (INTERVIEW MUST‑KNOW)

***

### ✅ 1️⃣ Choreography‑Based Saga (Event‑Driven)

No central controller.

    Order Service
      |
      | OrderCreated
      v
    Inventory Service
      |
      | InventoryReserved
      v
    Payment Service

### ✅ Flow with Failure

    Payment Failed
       |
       v
    Inventory Service → Release Stock
       |
       v
    Order Service → Mark Order Failed

✅ Pros:

*   Decoupled
*   Scales well

❌ Cons:

*   Harder to debug
*   Event explosion

✅ Interview line:

> “Choreography is simple but can become complex at scale.”

***

### ✅ 2️⃣ Orchestration‑Based Saga (Central Controller)

A **Saga Orchestrator** controls steps.

            Saga Orchestrator
                   |
            ----------------
            |       |      |
            v       v      v
         Order  Inventory Payment
         Service  Service   Service

### ✅ Flow

1.  Orchestrator → Create Order
2.  Orchestrator → Reserve Inventory
3.  Orchestrator → Process Payment
4.  On failure → Execute rollback steps

✅ Pros:

*   Easier monitoring
*   Central control

❌ Cons:

*   Orchestrator becomes critical

✅ Interview line:

> “Orchestration improves observability and error handling.”

# PART 2️⃣ Choreography vs Orchestration (VERY IMPORTANT)

## ✅ Choreography‑Based Saga

### 🔹 Concept

*   No central controller
*   Services communicate using **events**
*   Each service reacts to events and publishes the next event

### 🔹 Flow Diagram

    Order Service
      └─ OrderCreated →
             Inventory Service
                └─ InventoryReserved →
                       Payment Service

### ✅ Pros

*   Loose coupling
*   Highly scalable
*   No single point of control

### ❌ Cons

*   Harder to trace/debug
*   Event explosion as system grows

### ✅ Interview One‑Liner

> “Choreography is decentralized but harder to observe.”

***

## ✅ Orchestration‑Based Saga

### 🔹 Concept

*   A **Saga Orchestrator** controls the entire workflow
*   Services don’t talk to each other directly

### 🔹 Flow Diagram

    Saga Orchestrator
     ├─ CreateOrder → Order Service
     ├─ ReserveStock → Inventory Service
     └─ ProcessPayment → Payment Service

### ✅ Pros

*   Clear control flow
*   Easier monitoring & debugging
*   Centralized retry & compensation logic

### ❌ Cons

*   Orchestrator can become complex
*   Extra component to manage

### ✅ Interview One‑Liner

> “Orchestration improves observability at the cost of centralization.”

***

## 🔥 Comparison (Say This Verbally)

| Aspect        | Choreography     | Orchestration     |
| ------------- | ---------------- | ----------------- |
| Control       | Decentralized    | Centralized       |
| Communication | Events           | Commands          |
| Debugging     | Hard             | Easier            |
| Scalability   | High             | High (with care)  |
| Best For      | Simple workflows | Complex workflows |

✅ **Never say one is always better.**

# PART 3️⃣ Saga Pattern in .NET — CODE EXAMPLES

## ✅ Example Scenario

Order → Inventory → Payment

***

## ✅ 1️⃣ Choreography‑Based Saga (.NET Event Driven)

### 🔹 Event Definitions

```csharp
public record OrderCreated(Guid OrderId);
public record InventoryReserved(Guid OrderId);
public record PaymentFailed(Guid OrderId);
```

***

### 🔹 Order Service

```csharp
public class OrderService
{
    public void CreateOrder(Guid orderId)
    {
        // Save order
        Publish(new OrderCreated(orderId));
    }
}
```

***

### 🔹 Inventory Service

```csharp
public class InventoryService
{
    public void Handle(OrderCreated evt)
    {
        bool stockAvailable = ReserveStock(evt.OrderId);

        if (stockAvailable)
            Publish(new InventoryReserved(evt.OrderId));
        else
            Publish(new PaymentFailed(evt.OrderId));
    }
}
```

***

### 🔹 Payment Service

```csharp
public class PaymentService
{
    public void Handle(InventoryReserved evt)
    {
        bool paymentOk = ProcessPayment(evt.OrderId);

        if (!paymentOk)
            Publish(new PaymentFailed(evt.OrderId));
    }
}
```

***

### 🔹 Compensation Example

```csharp
public class InventoryService
{
    public void Handle(PaymentFailed evt)
    {
        ReleaseStock(evt.OrderId);
    }
}
```

✅ **Key Interview Explanation**

> “Each service reacts to events and performs local transactions.”

***

## ✅ 2️⃣ Orchestration‑Based Saga (.NET)

### 🔹 Saga Orchestrator

```csharp
public class OrderSaga
{
    private readonly IOrderService _order;
    private readonly IInventoryService _inventory;
    private readonly IPaymentService _payment;

    public async Task ExecuteAsync(Guid orderId)
    {
        try
        {
            await _order.Create(orderId);
            await _inventory.Reserve(orderId);
            await _payment.Process(orderId);
        }
        catch
        {
            await _inventory.Release(orderId);
            await _order.Cancel(orderId);
        }
    }
}
```

✅ Everything is visible in one place.
