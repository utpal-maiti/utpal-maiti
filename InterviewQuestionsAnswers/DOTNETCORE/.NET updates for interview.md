## ✅ Step 2: Define Interfaces (VERY IMPORTANT)

Interviewers love **interfaces + DI**.

```csharp
public interface IOrderService
{
    Task<Order> CreateOrderAsync(CreateOrderRequest request);
}

public interface IPaymentService
{
    Task<bool> ProcessPaymentAsync(Payment payment);
}

public interface IInventoryService
{
    Task<bool> ReserveStockAsync(List<OrderItem> items);
}
```

✅ Shows:

*   Loose coupling
*   Testability
*   Clean architecture

***

## ✅ Step 3: Domain Models

```csharp
public class Order
{
    public Guid Id { get; private set; }
    public List<OrderItem> Items { get; private set; }
    public OrderStatus Status { get; private set; }
    public decimal TotalAmount { get; private set; }

    public Order(List<OrderItem> items)
    {
        Id = Guid.NewGuid();
        Items = items;
        Status = OrderStatus.Created;
        TotalAmount = items.Sum(i => i.Price * i.Quantity);
    }
}
```

✅ Entity rules live **inside** the domain model.

***

## ✅ Step 4: Service Implementation

```csharp
public class OrderService : IOrderService
{
    private readonly IPaymentService _payment;
    private readonly IInventoryService _inventory;

    public OrderService(
        IPaymentService payment,
        IInventoryService inventory)
    {
        _payment = payment;
        _inventory = inventory;
    }

    public async Task<Order> CreateOrderAsync(CreateOrderRequest request)
    {
        var order = new Order(request.Items);

        var stockOk = await _inventory.ReserveStockAsync(order.Items);
        if (!stockOk)
            throw new Exception("Out of stock");

        var paymentOk = await _payment.ProcessPaymentAsync(
            new Payment(order.TotalAmount));

        if (!paymentOk)
            throw new Exception("Payment failed");

        return order;
    }
}
```

✅ Interview keywords:

*   **Single Responsibility**
*   **Dependency Injection**
*   **Async/await**

## ✅ Retries – What and Why

### 🔹 What is Retry?

*   502 / timeouts

❌ Not for:

# 1️⃣ How to Replay Events in Kafka

## ✅ What Does “Replay” Mean in Kafka?

**Replaying events** means:

> Reading **old messages again** from a Kafka topic to rebuild state, debug issues, or recover a service.

Kafka allows this because:

*   Messages are **immutable**
*   Stored for a **configurable retention period**
*   Consumers control **offsets**

✅ This is a **huge advantage** in Saga workflows.

***

## ✅ Ways to Replay Events (Interview‑Important)

### ✅ 1. Reset Consumer Offset (Most Common)

Each Kafka consumer tracks its **offset** (position in a partition).

To replay:

*   Reset offset to **earlier position**
*   Reconsume messages

📌 Conceptually:

```text
Offset = 0  ← replay everything
Offset = 120 ← replay from message 120
```

✅ Interview phrase:

> “Kafka replay is done by resetting consumer offsets.”

***

### ✅ 2. Use a New Consumer Group

Kafka stores offsets **per consumer group**.

✅ If you change the `GroupId`, Kafka treats it as a new consumer.

```csharp
GroupId = "inventory-service-replay"
```

➡️ Kafka starts reading from the beginning (depending on config)

✅ Interview line:

> “Starting a new consumer group is the easiest way to replay events.”

***

### ✅ 3. AutoOffsetReset = Earliest

```csharp
var config = new ConsumerConfig
{
    BootstrapServers = "localhost:9092",
    GroupId = "order-replay",
    AutoOffsetReset = AutoOffsetReset.Earliest
};
```

✅ Works **only if no committed offset exists**.

# 1️⃣ How to Handle Duplicate Messages in Kafka

## 🚨 Why Do Duplicate Messages Happen?

Kafka provides **at‑least‑once delivery** by default. Duplicates can occur due to:

*   Consumer crash before offset commit
*   Retry after transient failure
*   Rebalancing
*   Network timeouts

✅ **Duplicates are normal in Kafka systems**  
❌ Assuming “exactly once” everywhere is a common interview mistake

✅ Interview must‑say:

> “Kafka applications must be idempotent.”

***

## ✅ Core Strategies to Handle Duplicates

### ✅ 1. Idempotent Consumers (MOST IMPORTANT)

Ensure processing the same message **multiple times causes no side effects**.

### ✅ Approach

*   Use a **business key** (orderId, paymentId)
*   Store processed message IDs

```csharp
if (_processedMessageIds.Contains(messageId))
{
    return; // ignore duplicate
}

_processedMessageIds.Add(messageId);
```

✅ Interview line:

> “Idempotency ensures duplicate messages don’t cause duplicate state changes.”

***

### ✅ 2. Deduplication Table (Production‑Grade)

Store processed Kafka message IDs in DB (recommended).

```sql
ProcessedMessages
------------------
MessageId  |  ProcessedAt
```

```csharp
if (await db.ProcessedMessages.AnyAsync(m => m.MessageId == messageId))
    return;

await db.ProcessedMessages.AddAsync(new ProcessedMessage(messageId));
```

✅ Works across restarts ✅  
✅ Safe for distributed consumers ✅

***

### ✅ 3. Use Kafka Message Key Correctly

```csharp
Key = orderId.ToString();
```

✅ Guarantees:

*   Same orderId → same partition
*   Ordered processing
*   Easier idempotency

✅ Interview phrase:

> “Kafka keys help preserve ordering, not deduplication.”

***

### ✅ 4. Exactly‑Once Semantics (Advanced)

Kafka supports **producer EOS + transactions**, but:

*   Complex to manage
*   Not always required

✅ Interview answer:

> “Exactly‑once is useful, but idempotent consumers are simpler and safer.”
